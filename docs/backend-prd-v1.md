# 后端一期 PRD（v1）

本文定义旅行 AA 记账小程序的后端一期范围。  
与 [后端接口需求文档](/Users/boluobao/WeChatProjects/miniprogram-1/docs/backend-api-requirements.md) 配套使用。

## 1. 背景

前端当前已有首页、创建组队、旅行详情、成员管理、记账、结算等页面，但正式项目不会保留离线数据。  
一期目标是把这些页面背后的数据、登录、成员、账单和结算全部收口到后端。

## 2. 一期目标

1. 微信登录后先建立会话；创建组队或邀请认领后，能在对应旅行里定位到唯一的内建用户。
2. 内建用户的昵称、头像可持久化。
3. 能创建旅行、添加成员、记账、结算、标记已结清。
4. 微信绑定可换绑，且不影响当前旅行和历史旅行中的业务数据。
5. 前端生产环境只依赖后端，不再依赖本地 demo 数据。

## 3. 核心原则

### 3.1 用户分层

- `内建用户` 是旅行内的业务主体。
- `微信账号` 只是登录凭证和绑定入口。
- 同一微信账号可以在不同旅行里对应不同的内建用户，昵称和头像可以相同，但 id 不同。
- 旅行、成员、账单都归属旅行内建用户，不归属微信账号。
- 这里说的“用户”是旅行作用域里的角色，不是全局唯一账号。

### 3.2 绑定规则

- 一个微信账号在同一旅行里同一时刻只能绑定一个内建用户。
- 一个内建用户同一时刻只能绑定一个微信账号。
- 微信绑定不开放普通用户自助换绑；管理员可先解除某成员绑定，再让新的微信账号在当前旅行里绑定这个成员。
- 微信登录本身不自动复用其他旅行里的内建用户。
- 新旅行默认重新创建新的内建用户；邀请认领优先绑定当前旅行里尚未绑定的成员，若没有可认领成员，则按昵称和头像创建新的旅行内建用户并绑定当前微信账号。

### 3.3 资料规则

- 昵称、头像以内建用户资料为准；对用户可见的成员资料只有这两个字段。
- 微信昵称、头像只用于当前旅行内首次初始化或显式同步，不作为长期唯一来源。

## 4. 用户体系

### 4.1 内建用户

| 字段 | 说明 |
|---|---|
| id | 主键 |
| travel_id | 所属旅行 |
| nickname | 展示昵称 |
| avatar_url | 头像 |
| status | 启用 / 禁用 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

### 4.2 微信绑定

| 字段 | 说明 |
|---|---|
| id | 主键 |
| travel_id | 绑定所属旅行 |
| user_id | 绑定到哪个内建用户 |
| appid | 小程序 appid，用于区分不同微信应用 |
| openid | 微信登录标识 |
| unionid | 可选辅助标识，能拿到就存，不作为唯一标识 |
| bound_at | 绑定时间 |
| unbound_at | 解绑时间，换绑或解绑时记录 |
| last_login_at | 最近登录时间 |
| is_active | 是否当前有效绑定 |

### 4.3 关键行为

- 微信登录：只建立登录态，不直接创建或复用任何旅行内建用户。
- 创建新旅行：创建该旅行里的新内建用户，并绑定当前微信账号。
- 进入已有旅行：若当前旅行已有绑定，返回对应内建用户；若无绑定，走邀请认领流程，或在无可认领成员时创建新成员并绑定。
- 换绑：先解除当前旅行里的旧绑定，再绑定到目标内建用户。
- 不同旅行里的内建用户不会自动继承。

## 5. 一期范围

### P0 必做

1. 微信登录、旅行列表、创建旅行。
2. 旅行内绑定、获取当前内建用户、修改昵称和头像。
3. 旅行详情。
4. 添加成员。
5. 新增花销、账单列表、账单明细。
6. 结算结果和结算明细。
7. 标记已结清。

### P1 应做

1. 头像上传。
2. 成员认领 / 清除认领。
3. 成员禁用记账。
4. 成员移出组队。
5. 微信分享卡片加入。
6. 账单编辑 / 删除。
7. 旅行基础设置保存。

### P2 不纳入一期

1. 猫猫家庭。
2. 预收款 / 猫猫基金独立台账。
3. 归档 / 删除旅行。
4. 分类字典。
5. 扫码加入的完整扩展版。

## 6. 核心流程

### 6.1 登录与绑定

1. 小程序调用 `wx.login`。
2. 后端换取微信身份。
3. 后端返回登录态和是否已有旅行绑定，不返回全局业务用户。
4. 前端按入口分流：无旅行时先创建组队；有旅行时可进入首页列表；分享卡片进入时走邀请认领。
5. 只有进入具体旅行后，前端才通过旅行作用域接口拿到“当前微信在这次旅行里是否已绑定内建用户”的结果。

### 6.2 创建旅行

1. 用户在创建组队页选择头像和昵称。
2. 创建完成时，后端根据这次选择创建一个新的旅行内建用户。
3. 当前微信账号绑定到这个新建的旅行内用户。
4. 后端创建旅行，当前用户自动成为管理员。
5. 同行成员写入成员表。
6. 后端返回旅行详情和分享卡片参数；前端可先进入详情，之后首页列表通过绑定关系展示这次旅行。

### 6.3 微信分享卡片进入

1. 用户通过微信分享卡片进入某次旅行。
2. 后端展示这次旅行里尚未绑定微信的成员。
3. 若有可认领成员，用户从未绑定成员中选择一个完成绑定。
4. 若没有可认领成员，前端展示头像和昵称输入，用户点击加入后，后端按这次填写的新资料创建新的旅行内建用户并绑定当前微信账号。
5. 绑定成功后，这个成员就对应到当前微信账号，且只影响这次旅行。

### 6.4 添加成员

1. 管理员在旅行详情里添加成员。
2. 后端创建或补全该旅行里的对应内建用户。
3. 成员先可不绑定微信，后续可在本旅行里认领。
4. 成员绑定的解除由管理员发起，解除后成员回到未绑定状态，后续可被新的微信账号认领。

### 6.5 记账

1. 选择付款人。
2. 选择参与分摊成员。
3. 后端保存账单和分摊明细。
4. 记账结果立即影响旅行汇总、账单列表和结算。

### 6.6 结算与结清

1. 后端根据账单计算净额和转账建议。
2. 前端展示成员净额和转账关系。
3. 结算明细要能看到每个成员的垫付、应摊和来源账单。
4. 管理员点击结清后，旅行状态改为已结清。

## 7. 数据库

一期后端数据库使用 `MySQL`，最少包含这些表：

1. `users`
2. `wechat_bindings`
3. `travels`
4. `travel_members`
5. `expenses`
6. `expense_shares`
7. `settlements`

补充建议：

- `travel_members` 可预留 `can_bookkeep`，用于成员禁用记账。
- `travel_members` 不必单独维护“认领状态”，认领是否成立可由当前旅行里的 `wechat_bindings` 是否激活判断。
- `users` 和 `wechat_bindings` 都要带 `travel_id`，以便同一微信账号在不同旅行里拥有不同内建用户。
- 账单最终归属应落到旅行内建用户 id。

## 8. 验收标准

1. 任意微信账号在同一旅行里只会落到一个内建用户。
2. 同一个微信账号可以在不同旅行里对应不同内建用户。
3. 用户头像和昵称修改后刷新不丢。
4. 旅行、成员、账单列表、账单明细、结算汇总与结算明细均来自后端。
5. 创建组队、首页列表、旅行详情到结算页能跑通完整闭环。
6. 线上数据不依赖本地存储兜底。
